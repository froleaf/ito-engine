在不同的目标下，指标体系有不同的拆解逻辑。
通常我们在拆解团队okr、测算业务季度/年度目标、以及从数据采集层开始搭建完整业务指标面板时，都会涉及到数据指标体系的构建。
增长黑客教我们，先找到北极星指标，然后向下进行多层拆解。
但在不同的目标上，指标体系的拆解逻辑也会有所区别。

业务指标拆解
第一种场景通常出现在一个业务团队（尤其是增长团队）在进行okr拆解和业务数据追踪的时候。
面板需要展示什么数据？团队中每个人需要关注什么数据？如何将数据变为指令或动作？
这些问题是该场景下的核心问题。
我们从“这些人是谁”、“他们看数据的目标是什么”、“他们要看什么数据”这三个方面来展开。

1. 投资层
  1. 投资层，即手握资源的人。可以指投资方、对很多业务团队来说也可能就是他们的高层领导。
  2. 对于投资层，他们看数据的目标很大程度上是为了判断“是否要投入更多的资源”。
  3. 投入资源一般有几种原因：有利益、存在长期利益、这件事情与价值观一致。对应的关注点会在盈利数据、用户规模数据、以及与业务/产品本身的价值点相关的指标（通常是北极星指标）。
  4. 另外，投资层没有时间看太多的细节内容，信息的分层与获取效率在这里非常重要。在这一层，数据通常以月聚合，观测数据的角度以按月的切片数据或是 cohort 数据为主。
    1. 按月的切片数据：这里主要是和漏斗数据区分开。例如我们在讨论活跃用户付费率的时候，上个月的切片数据是指「上个月付费用户/上个月活跃用户」；而漏斗数据是指「上个月活跃用户中的付费用户/上个月活跃用户」。两种统计口径会由于各种业务细节而出现差异。前者关注的是业务/产品的整体情况，后者关注的是用户的生命周期。
    2. cohort数据：当投资层将视角转移到用户身上时，通常会关注cohort数据。例如使用用户的注册年月作为分批依据，观测每一批用户在接下来每个月的关键行为表现，来了解用户的生命周期以及产品在月与月间的表现是否维持稳定或有明显的提升。

2. 业务决策层
  1. 业务决策层是指导业务优化方向、并对业务结果负责的人。很多时候也是这个角色在主导业务指标的拆解。
  2. 对于决策层，他们看数据的目标是为了掌握业务的发展方向并在适当的时候给出优化与调整的方向与策略。
  3. 业务通常是依存于“需求”的，所以了解业务/产品与用户的关系非常重要。决策层关注点有两类，一类是用户的分布情况，另一类是模块与模块之间的连接与协调。
    1. 用户分布情况可以通过用户标签或分层来实现观测。比如对新老用户分层来进行分生命周期的行为追踪和策略制定；对用户的活跃程度进行分层来了解产品的黏性；对用户的付费能力进行分层（如RFM模型）来了解用户付费能力的分布，以优化营销和定价策略。决策层借此了解自己业务/产品的整个用户盘子的现状，找到用户真正的痛点或是卡点，校正业务的发展方向，找到突破口。
    2. 小到营销组件与落地页链路优化、大到多方合作的业务调整，洗牌效果的案例比比皆是。有时候调整了广告的形式吸引更多人进入落地页，落地页的付费率却变低了，最终整体ROAS基本没有提升。而由多个模块连接构成的业务，也会出现这个模块业绩变好的同时，下游模块业绩却变差了。为了了解一个策略真实的有效性，同时协调模块与模块之间的利益冲突，决策者需要获得完整的业务链路数据。
  4. 决策层还有一个关键的动作，就是监测并规避大的风险。所以他们需要对于投资层与决策层指标的波动有一定的敏感性，指标监测功能对他们通常是很有用的。


3. 执行层
  1. 执行层是在既定的业务方向上进行方案落地的人。他们随时跟进自己所在模块的情况，以进行落地方案的优化调整与迭代，并将发现的问题向上汇报。
  2. 对于执行层，他们看数据的目标是为了得到方案执行结果的效果反馈，来帮助他们迭代新的方案。
  3. 执行层关注流入自己模块的用户的具体行为（例如用户路径或是节点的转化/流失率），来了解方案与用户的契合度，辅助下一轮的方案迭代；漏斗是非常常用的工具。有时候他们也会关注不同类型的用户在对应模块的表现，来了解迭代的优先级；或发现显著的问题与缺陷向上汇报。
  4. 数据面板对于执行层有一个额外的作用，就是周报/月报以及活动日报的撰写与汇报。这种周期性的模版汇报可以通过特定的数据面板来达到效率的提升。


业务目标测算
由于在不同的业务领域，测算方法各有不同；这里只简单地聊一下基本原则。
业务目标测算通常出现在两种场景：一种是业务负责人先确定了业务总目标后，向下进行下层的目标与指标拆解；另一种则是基于对业务已有的认知和数据表现，向上制定下一周期的业务目标。
无论是自上而下还是自下而上，测算的结果都是向北极星指标负责；所以最重要的是测算的准确性和可操作性。

第一步：将北极星指标拆解为，由可操作、可落地的变量组成的公式。
- 在业务层面，要保证拆解结果对北极星指标负责且可以被落地执行：
  - 首先需要保证该公式在业务上的可解释性，即保证公式的业务逻辑是正确的；通常会拆解为用户规模X转化率之类的模式；
  - 保证拆解出的每一个变量都是可以被业务同伴清晰理解和感知的；当用户的行为特征差异太大时（例如新用户与老用户），可以直接在公式中对用户进行分层来分开考虑和测算；
  - 在既定的组织架构下，尽量考虑到每一个组织的责任边界，来找到并确定对应的变量；这样可以帮助我们降低遗漏的可能性；
  - 尽量减少变量之间的依赖关系，以降低公式的复杂度。
- 在数据层面，为了提升测算的准确度，变量需要至少满足以下两个要求之一：
  - 变量本身可控。例如获客量，它对于业务来说是可控的，可以通过策略进行调整；
  - 变量的波动范围不大，有长期的稳定性，可以视为固定值。例如特定平台上特定领域广告投放的CAC在短期内的波动不大，通常可以视为固定值；其长期的变化可以通过后期进行数据追踪与校正来适应。除了利用经验以外，也可以通过对指标过去的数据表现进行监测，来确定它的波动情况。
  - （可被操作的变量会成为下一层的目标指标；稳定变量则暂时作为常量来辅助测算）

第二步：了解公式中指标与指标之间存在的依赖关系，标识风险并跟进。
- 业务中各个指标间可能会有难以拆开的依赖关系。例如在金融产品中，假设「盈利 = 募集资金量*平台收益率」。平台收益率受到其抽成比的影响，募集资金量受到用户收益率的影响。这时，如果采用调整抽成比的策略来提升平台收益率，就可能会导致降低用户收益率，进而影响到募集资金量。
  - 当这种影响可以通过公式进行建模时，我们可以直接通过调参的方式得到最佳策略；
  - 当这种影响难以量化时，则需要记录依赖关系与可能的风险，持续关注与跟进。

第三步：基于对公式中各个变量当前状态的掌握或近似，进行业务目标测算。
- 在自上而下的测算任务里，北极星指标的期望是确定的。通过对后续用户规模的测算，平衡各业务指标的增长幅度期望，并留出容纳风险的空间，来拟定下层各模块的业务目标；
- 在自下而上的测算任务里，北极星指标的期望是不确定的。同样可以通过对后续用户规模的测算，结合各业务指标的期望增长幅度，来给出整个业务的增长目标。
- 单独说一下用户规模测算：用户规模受到新增和留存的影响，处于不同生命周期的用户在留存率上的表现也有显著不同。通过了解不同生命周期用户的留存率，来构建留存带来的用户规模；再叠加上新增的用户量，来测算N个月后的总用户规模。
  - 例如新用户的次月留存率为20%，第三月留存率为12%，第四月留存率为8%……；从第8个月起，留存率便趋于稳定在3%上下，后续的降幅可以忽略不计。
  - 那么我们在测算下个月的用户规模时，可以使用过去每个月的新用户作为基数，来统计下个月的留存人数；再叠加下个月预计的新增用户数，得到测算的用户规模数。

第四步：按阶段对公式的各个变量与常量指标进行追踪与校正。
- 无论是业务本身、用户心智、还是社会与市场环境都在不断的变化，短期稳定的指标长期下来可能会发生很大的变动，而扭曲了测算效果；为了追踪目标的完成情况与风险，需要及时调整与更新测算细节。

业务数据基建
如果说前面两个场景以业务框架的思考居多，那么业务数据基建则是从数据如何落地的角度出发的。尤其对于一个全新的业务，它会成为数据采集方案的基本依据。
业务数据基建的关键逻辑是业务框架梳理+优先级确定。在实际的落地过程中，通常不会有足够的资源把方方面面的数据都采集到，如果想要一次推行一个大而全的工程，很容易以失败告终；所以「优先级」的确定尤为重要。
当然，不推行大而全的工程，并不意味着就不用进行一个大而全的框架梳理。框架梳理的完整性是正确确立「优先级」的先决条件。

我们从三个角度进行框架梳理：
1. 业务模块 x 增长与营收与北极星指标
  1. 增长指标：增长指标与用户规模相关。我们可以直接套用AARRR模型来确定增长相关的指标，即「获客、激活、留存、付费、传播」。用我比较熟悉的app业务举例：在实际操作中，我通常会用「新增、活跃、留存、付费、传播」作为增长指标的大纲。
    1. 其中“新增”部分关注获客链路上的几个卡点的数据：安装--启动--注册成功--核心功能首次使用；
    2. “活跃”部分拆为新老用户进行观测，关注用户打开app以及使用功能的次数、频率和时长；
    3. “留存”部分拆为新老用户进行观测，关注用户的短期与长期留存率和流失率，同时关注整体流量的流入和流失情况；
    4. “付费”部分通常关注ARPU与ARPPU、付费人数/订单数/金额、复购与续订情况等指标。
    5. “传播”链路并不是大部分app都有认真搭建的，这里就不进行描述了。
  2. 营收指标：GMV；
  3. 北极星相关指标：北极星指标涉及产品的价值本身。
    1. 在有些业务上（如电商），北极星指标GMV和营收指标是一致的；
    2. 而在其他业务中，通常有其各自特有的重要指标。例如教育业务会关注“出勤率”和“完课率”，搜索业务会关注“搜索次数”，社交业务会关注“互动数据”，社区业务会关注“UGC与时长数据”等等。
  4. 业务模块交叉各类指标：
    1. 以业务整体作为最高层，逐层向下梳理业务模块，直到最小的业务单元；
    2. 标注出核心业务模块与非核心业务模块；
    3. 将各个业务模块与上面列举的指标进行交叉；
    4. 优先级的划分通过模块直观地展示出来：其中业务整体指标为t0优先级，第二层的核心业务模块为t1优先级，第二层非核心模块及第三层核心模块为t2优先级……；也可以结合业务需求对优先级的细节进行决策。

2. 用户生命周期梳理
  1. 统一对于用户口径定义：例如在app类产品中，会有设备标识、用户id（注册后）；在基于微信生态的业务中，会有openid和unionid……这些都可能是导致最终产出的指标口径不同的原因；
  2. 通过用户生命周期梳理补全遗漏的关键事件。

3. 维度模型：通过维度模型补全遗漏的维度，方便数据分析时的分类与下钻
  1. 通用维度，如版本号、设备类型、网络等；
  2. 不同业务中通常会有特定的维度，如教育业务中关于班级和老师的属性、社区业务中关于UGC的属性等；
  3. 与用户相关的重要属性，如会员类型等。
