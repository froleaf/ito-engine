先来整理一下最近遇到的小朋友们困扰的问题。
首先数据组真的最苦最没产出，一定要找到能够衡量价值的标准；以及中台如果自己都不重视长期价值和系统稳定性，就更不会有其他人帮你思考这些问题。保证系统的健壮性很重要，不要只是去关注眼前的业务价值，不然何必要成立中台呢？
若璇这边有遇到四个方面的问题，关于【如何汇报】、【如何写分析文档】、【如何做方案】、【如何和要面子的人交流】。
1. 对于汇报，可以基于场景有多种偏向。1）权力明显的环境，要学会「汇报工作量」并「上升价值」；尤其是那种管理者基本不看内容，就想保证你是在工作的，「汇报工作量」会成为非常有用的一种方式。同样，这类管理者不愿意自己去抽象出来结论，所以他们只能看到最基础的执行动作的结果，稍微有一些复杂的事情他们就会失去价值锚点。所以他们需要你帮他们聚焦和上升价值，简称画饼。不要只老老实实给结果。2）踏实且可信赖的环境，汇报会成为向上索要资源的主要途径。3）面向普通管理者，日常的进展汇报用文字写清楚【进度/计划/风险】同步即可。
2. 分析文档的前中后：确定分析目标--框架、分析、结果--形成业务洞察。通常的新人只会有「中的中：分析」；但很快能够掌握「中」的全部内容。缺头少尾是很常见的，基本是由于对“了解业务”的重要性认识不够。1）关于头：很多分析者是没拆分清楚业务目标到分析目标的路径的，可能就直接开始取数了。这会导致分析过程反复改，最后的结果依然作用不大。2）关于尾：尾比头更难，也更重要。最初的分析师会完全不在意业务洞察的部分，一方面他们认为这种抽象像是空想或是废话，不像分析过程这样具有技术性和科学有效；另一方面他们对着这些分析结果，很难抽象出来业务结论和洞察，这是一个需要真正的站在业务视角跟业务方一起磨练过，才能够具备的能力。缺少业务洞察，会显得文档里只有分析没有真实的业务结论或者推论。但其实只有结论和推论才会真正推动事情发展，包括得到有效信息、给出下一步的行动引导、得到需要继续验证的思路，等等。
3. 方案的前中后：框架-内容（含风险预案）-复盘。新人接到需求时，通常会直接写方案内容，就会显得很散，没有主干。1）头是做方案时最关键的部分。首先对目标和要做的事情（5w1h）有大体的认知之后，梳理出来一条时间线图。因为时间永远是方案的决定性资源成本。在这个基础上再去考虑什么事情需要被决策，决策背景等。2）虽然对单个方案来说，头比尾关键；但从长期价值来说，尾更重要，也更难。因为方案都有可能失败，持续性地用正确的方法做事才能导向成功。而复盘才是对长期的风险意识和经验的积累。
4. 如何交流：现实中真的太多死要面子唧唧歪歪的人了（尤其男的）。当然首先自己要成为一个有专业性的人，这是第一步。无论是能力的专业性还是履历的专业性。越是唧唧歪歪的人，其实越是迷信权威，他们处在纵向关系中。他们有情绪、有你无法理解的逻辑，但你不要陷进去了，不要被他们的逻辑或者他们的情绪绕进去。只要你在专业性上强势地压过他们，他们就会自有一套说服自己的逻辑，也会处理好自己的情绪。就是这么简单。

大多数人对于数据分析的能力模型都不太清楚。我总结下来的话分为5个方面。
1. 业务知识--软技能（对实际业务的理解与经验） + 硬技能（对指标的概念与拆解能力）
2. 数据工程--数据库结构设计、表结构设计、存储技术（面向事务与面向分析）
  1. 正推：如何从产品和数据库规范设计数据表
  2. 反推：如何从最终的数据呈现反推底层需要的数据节点和维度
3. 技术能力--SQL、python、前后端与测试的基本概念
4. 可视化--工具的熟练度 + 设计意识与能力（信息传递能力--布局与图表设计）
5. 业务分析--统计学意识 + 分析框架 + 文档能力（信息传递能力--布局与文字功底）
  1. 数据分析文档的标准
  2. 横向用户生命周期 x 纵向分维度；
  3. 周期内时间尺度 x 周期间同比环比。
