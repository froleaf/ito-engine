# ITO 协作启动文档

## 一句话定位

ITO 是一个 AI 驱动的认知引擎——帮用户把散落的阅读、思考和对话编织成带有个人印记的知识图谱，并推动它们走向输出。用得越久越像你，最终成为你的认知分身。

## 核心价值观

**AI 的价值不是替代你的思考，而是释放你去思考。**

基于 Kahneman 的 System 1/2，我们区分两种心智活动：
- **功利的系统一**：为外人负责的事务型劳动（填表、格式化、重复沟通）
- **热爱的系统一**：你与世界连接的触手（手指在琴键上的触感、亲手做数据分析的判断力积累）

AI 应该替代前者，保护后者。大部分人用 AI 的方式是把自己变成老板和 AI 之间的传话筒——ITO 不做这件事。

**四层价值主张：**
1. 替代功利的系统一，解放人
2. 激发系统二——苏格拉底对话 + 三重触手（世界知识 / 外部信息 / 记忆）
3. 可视化成长——大脑可视，对抗"你会被替代"的焦虑
4. 保护热爱的系统一——不替代你与世界的触手，让你有更多时间去触摸

## 当前架构

### 技术栈

- **数据层**：JSON-LD 统一知识图谱（`ontology/ontology.jsonld`）
- **记忆层**：JSONL 日志（session_memory、plan_memory、milestone_memory、todo）
- **交互层**：Markdown Skill 系统（`.claude/skills/*/SKILL.md`）
- **可视化**：D3.js 力导向图（自动从图谱生成）
- **运行时**：Claude Code CLI（无自建后端，AI 直接读写文件系统）

### 为什么是 JSON-LD 而不是数据库？

**图范式复兴论**（这是我们的架构哲学核心）：

被低估的不是图数据库技术，而是"对象+带属性的边"这个建模范式。

两条核心理念：
1. **连接本身和对象一样重要**——边是一等公民，可以有类型、属性、权重
2. **信息复杂性超越二维**——关系型数据库的 JOIN 是固定深度的，图的遍历是动态的

为什么现在是时候：
- AI 天然有世界知识，高校规模化已产生统一的术语体系——这就是现成的 ontology 骨架
- AI 能自动完成实体构建——构建瓶颈被解决
- JSON-LD 文件比 Neo4j 更适合个人场景——更轻、AI 可直接读写、不需要基础设施

**与 RAG 的关系**：不是替代，是互补。RAG 让 AI 找到相关的东西，JSON-LD 让 AI 知道这些东西之间是什么关系。RAG 的根本问题是检索和推理断裂——它把相关片段堆在一起让 LLM 自己理出头绪，但不知道哪个是前提、哪个是结论。图范式中关系是显性的，AI 不需要猜。

**健康 Coach 场景验证**：RAG 输出最不稳定（概率性召回→矛盾建议=安全风险），完全结构化策略库凸显不出 AI 价值（像查表），图范式给 AI 装上护栏但不锁死——沿图遍历的路径是确定性的、可审计的。滑雪 AI Coach 已是原型验证。

## 愿景：I → T → O 全链路

ITO 这个名字本身就是路线图。当前的核心是 **T（Think）**，接下来要向两端延伸：

```
┌─────────────────────────────────────────────────────────────┐
│                     ITO 全链路愿景                           │
│                                                             │
│   I（Input）          T（Think）           O（Output）       │
│   信号捕手             ITO 核心引擎         认知代理          │
│                                                             │
│   信息源收集           苏格拉底对话         大脑模式进化       │
│   智能筛选             知识提取+图谱        个性化内容生成     │
│   个性化阅读           跨域发现             多格式输出        │
│   预期构建             思维脚本编译         发布管道           │
│                                                             │
│   ← 千人千面的认知适配    认知图谱    用你的声音说话 →        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### I（Input）— 信号捕手

信号捕手是一个已有的原创框架，核心价值观："在满足诉求的基础上，潜移默化地提升认知"。

**三大模块：**
1. **用户状态解析器**：认知带宽（当下能量）× 结构分辨率（认知复杂度）
2. **内容认知评估器**：抽象阶梯（L1 故事 → L5 元逻辑）× 认知密度（L1 噪音 → L5 智慧）
3. **认知-内容交互模型**：5×5 矩阵，内容与用户状态匹配

**新增设计（2026-03 扩展）：**
- **预期构建**（第零层）：基于预测编码理论，在呈现内容前先为用户建立认知锚点。有用户图谱的 AI 能从意图构建预期（"这篇和你做的事很像但方向不同"），通用 chatbot 只能从摘要构建
- **离心分层**：内容 AI 应是离心机而非压缩机——分离 L1-L5 各层保留特征，而非全部碾平为 L3 摘要
- **按状态呈现**：根据用户认知带宽决定呈现哪层

**与 ITO 的千人千面打通**：信号捕手的用户状态解析器 + ITO 的认知图谱 = 真正的个性化。不是"千人千面的推荐"（抖音），而是"千人千面的认知适配"——同一篇内容，根据你是谁来决定怎么给你讲。

> 详见：`output/deposits/td_signal_catcher.md`、`output/deposits/td_content_ai_design.md`

### T（Think）— 当前核心

已基本跑通。核心能力：
- 苏格拉底式对话（/chat）
- 自动知识提取和图谱更新
- 跨域关联发现（CrossDomainLink）
- 个人思维脚本编译（/compile-thinking）
- PDCA 精力分配和周度复盘

待优化：
- AI 模型兼容性（当前绑定 Claude Code CLI）
- 图谱查询性能（grep 方案在节点增长后会遇到瓶颈）
- 多用户支持

### O（Output）— 认知代理进化

当前 `/brain` 是第一版——以用户的知识和思维方式对外输出，严格只读。

进化方向：
- **多场景人格/语气适配**：工作、朋友、社交媒体等不同语境
- **长文生成**：从 Deposit 中编译完整文章，而非只做对话式输出
- **多格式**：文章、PPT 大纲、视频脚本、教学材料
- **发布管道**：直接连接发布平台（公众号、小红书等）
- **协作输出**：多人的认知图谱交叉，产出协作内容

## 需要协作推进的方向

### 近期（建立基础）

1. **AI 模型兼容**：当前绑定 Claude Code CLI，需要抽象出模型接口层，支持其他模型（GPT、Gemini 等）
2. **Skill 系统工程化**：当前 Skill 是 Markdown 文件被 AI 读取执行，需要评估是否需要代码化的执行层
3. **多用户架构**：当前单用户单文件夹，多用户需要考虑图谱隔离和共享

### 中期（双向延伸）

4. **信号捕手产品化**（I 端）：信息源接入（RSS、社交媒体、Newsletter）→ 离心分层 → 预期构建 → 个性化呈现
5. **输出能力进化**（O 端）：从对话式 brain 到结构化内容生成，场景化语气，发布管道

### 远期（生态）

6. **图谱互联**：不同用户的图谱之间能否发现连接？（类似学术引用网络）
7. **垂直场景拓展**：教育（小学生学习路径）、健康（Coach 知识库）、项目管理等
