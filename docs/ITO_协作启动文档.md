# ITO 协作启动文档

## 30 秒版本

- **是什么**：AI 驱动的个人认知引擎——把阅读、思考、对话编织成知识图谱，再用它为你输出
- **为什么这么做**：AI 应该释放人去思考，而不是把人变成传话筒；软件应该轻量、本地、属于个人，而不是绑在中心化平台上
- **怎么做到的**：没有后端、没有数据库、没有账号——JSON-LD 文件 + Markdown Skill + AI 运行时，整个系统几乎没有自建工程代码
- **现在到哪了**：T（Think）基本跑通，288 个知识节点、46 篇思考沉淀；I（Input）和 O（Output）在设计中
- **需要什么帮助**：AI 模型兼容、前端触达更多人、信号捕手产品化、输出能力进化

## 一句话定位

ITO 是一个 AI 驱动的认知引擎——帮用户把散落的阅读、思考和对话编织成带有个人印记的知识图谱，并推动它们走向输出。用得越久越像你，最终成为你的认知分身。

## 核心价值观

**AI 的价值不是替代你的思考，而是释放你去思考。**

基于 Kahneman 的 System 1/2，我们区分两种心智活动：
- **功利的系统一**：为外人负责的事务型劳动（填表、格式化、重复沟通）
- **热爱的系统一**：你与世界连接的触手（手指在琴键上的触感、亲手做数据分析的判断力积累）

AI 应该替代前者，保护后者。大部分人用 AI 的方式是把自己变成老板和 AI 之间的传话筒——ITO 不做这件事。

**四层价值主张：**
1. 替代功利的系统一，解放人
2. 激发系统二——苏格拉底对话 + 三重触手（世界知识 / 外部信息 / 记忆）
3. 可视化成长——大脑可视，对抗"你会被替代"的焦虑
4. 保护热爱的系统一——不替代你与世界的触手，让你有更多时间去触摸

> 详见：`output/deposits/td_ito_product_philosophy.md`

## 技术哲学：去工程化与个体主权

核心价值观回答的是"AI 应该怎么对待人"，技术哲学回答的是"AI 时代软件应该长什么样"。两者共同构成 ITO 的底层信念——**个体主权**。

### 四个设计选择

| 选择 | 主张 | 在 ITO 中的体现 |
|------|------|-----------------|
| **图/网状存储** | 知识的组织权归个人，不是推荐算法替你决定 | JSON-LD 统一图谱（具体方案见下方"为什么是 JSON-LD"） |
| **文件本地化** | 数据所有权归个人，隐私不是可交换的商品 | 全部数据存本地文件系统，无云端依赖 |
| **去工程化** | AI 让自建工程代码不再必要 | 整个系统几乎没有自建代码，Skill 是自然语言的 Markdown |
| **个体主权** | 不需要工程团队也能实现想法 | 一个人 + AI + 文件系统 = 完整的认知引擎 |

### 架构倒置

传统软件工程：用代码搭架构，用配置处理细节。

ITO 的方式：**自然语言做宏观架构，代码只做微观 IO**。AI 读 Markdown 就能执行复杂的认知工作流，需要代码的地方（可视化、数据转换）范围小、AI 容易写好。这不是偷懒，而是一种更适合 AI 时代的架构分层策略。

**必须正视的依赖**：去工程化不是消灭了代码，而是把代码从"自己写"变成了"AI 运行时提供"。过去的范式是编程语言开源但编程门槛高，现在的范式可能变成 AI 技术闭源但上层门槛变低。ITO 依赖 Claude 作为运行时——如果 AI 能力退化或 API 断供，系统就会受影响。这是去工程化的前提条件，也是推动 AI 模型兼容（见协作方向）的核心动机。

### 工程复杂性三层膨胀模型

工程复杂性的膨胀往往不是技术需求驱动的，而是与**企业想控制什么**高度相关：

```
第一层：开源分发
  skill + 知识大脑 → 用户设备
  零工程复杂性，AI 本地执行

第二层：skill 私有化
  企业保留 skill → 需要服务器
  并发、实时性、可用性问题出现

第三层：数据中心化
  用户数据也存在服务器 → 规模爆炸
  隐私、合规、安全问题全来了
```

控制越多，工程越重。ITO 选择第一层。

**反面视角**：企业选择第二层、第三层不全是控制欲——质量保证（开源 skill 被改坏了谁负责）、合规责任（医疗 Coach 出了问题谁担责）、商业可持续性（纯开源怎么养团队）都是合理动机。但我们认为，这些问题可以通过知识产权保护、数据交易模型、本地加密分发等方式解决，而不必走向中心化。如何在不中心化的前提下解决这些问题，本身就是一个值得探索的技术和商业方向。

### 前端策略：Builder + Reader 双模式

前端降低使用门槛（地板），但也框住了用户能做的事（天花板）。前端本质上也是一种设计约束——设计者替用户决定了"你能做什么"。CLI 也有约束，但它的约束是可编程的、弹性的。

ITO 的理想状态：
- **Builder 模式**（CLI）：工程师思维，可以改 skill、调图谱、加逻辑
- **Reader 模式**（前端）：降低门槛，触达更多人，营造适合写作的环境
- 两者共存，而不是前端取代命令行

## 当前架构

### 技术栈

- **数据层**：JSON-LD 统一知识图谱（`ontology/ontology.jsonld`）
- **记忆层**：JSONL 日志（session_memory、plan_memory、milestone_memory、todo）
- **交互层**：Markdown Skill 系统（`.claude/skills/*/SKILL.md`）
- **可视化**：D3.js 力导向图（自动从图谱生成）
- **运行时**：Claude Code CLI（无自建后端，AI 直接读写文件系统）

### 为什么是 JSON-LD 而不是数据库？

这是上述技术哲学中"图/网状存储"选择的具体展开。

**图范式复兴论**（这是我们的架构哲学核心）：

被低估的不是图数据库技术，而是"对象+带属性的边"这个建模范式。

两条核心理念：
1. **连接本身和对象一样重要**——边是一等公民，可以有类型、属性、权重
2. **信息复杂性超越二维**——关系型数据库的 JOIN 是固定深度的，图的遍历是动态的

为什么现在是时候：
- AI 天然有世界知识，高校规模化已产生统一的术语体系——这就是现成的 ontology 骨架
- AI 能自动完成实体构建——构建瓶颈被解决
- JSON-LD 文件比 Neo4j 更适合个人场景——更轻、AI 可直接读写、不需要基础设施

**与 RAG 的关系**：不是替代，是互补。RAG 让 AI 找到相关的东西，JSON-LD 让 AI 知道这些东西之间是什么关系。RAG 的根本问题是检索和推理断裂——它把相关片段堆在一起让 LLM 自己理出头绪，但不知道哪个是前提、哪个是结论。图范式中关系是显性的，AI 不需要猜。

**健康 Coach 场景验证**：RAG 输出最不稳定（概率性召回→矛盾建议=安全风险），完全结构化策略库凸显不出 AI 价值（像查表），图范式给 AI 装上护栏但不锁死——沿图遍历的路径是确定性的、可审计的。滑雪 AI Coach 已是原型验证。

## 愿景：I → T → O 全链路

ITO 这个名字本身就是路线图。当前的核心是 **T（Think）**，接下来要向两端延伸：

```
┌─────────────────────────────────────────────────────────────┐
│                     ITO 全链路愿景                           │
│                                                             │
│   I（Input）          T（Think）           O（Output）       │
│   信号捕手             ITO 核心引擎         认知代理          │
│                                                             │
│   信息源收集           苏格拉底对话         大脑模式进化       │
│   智能筛选             知识提取+图谱        个性化内容生成     │
│   个性化阅读           跨域发现             多格式输出        │
│   预期构建             思维脚本编译         发布管道           │
│                                                             │
│   ← 千人千面的认知适配    认知图谱    用你的声音说话 →        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### I（Input）— 信号捕手

信号捕手是一个已有的原创框架，核心价值观："在满足诉求的基础上，潜移默化地提升认知"。

**三大模块：**
1. **用户状态解析器**：认知带宽（当下能量）× 结构分辨率（认知复杂度）
2. **内容认知评估器**：抽象阶梯（L1 故事 → L5 元逻辑）× 认知密度（L1 噪音 → L5 智慧）
3. **认知-内容交互模型**：5×5 矩阵，内容与用户状态匹配

**新增设计（2026-03 扩展）：**
- **预期构建**（第零层）：基于预测编码理论，在呈现内容前先为用户建立认知锚点。有用户图谱的 AI 能从意图构建预期（"这篇和你做的事很像但方向不同"），通用 chatbot 只能从摘要构建
- **离心分层**：内容 AI 应是离心机而非压缩机——分离 L1-L5 各层保留特征，而非全部碾平为 L3 摘要
- **按状态呈现**：根据用户认知带宽决定呈现哪层

**与 ITO 的千人千面打通**：信号捕手的用户状态解析器 + ITO 的认知图谱 = 真正的个性化。不是"千人千面的推荐"（抖音），而是"千人千面的认知适配"——同一篇内容，根据你是谁来决定怎么给你讲。

> 详见：`output/deposits/td_signal_catcher.md`、`output/deposits/td_content_ai_design.md`

### T（Think）— 当前核心

已基本跑通。核心能力：
- 苏格拉底式对话（/chat）
- 自动知识提取和图谱更新
- 跨域关联发现（CrossDomainLink）
- 个人思维脚本编译（/compile-thinking）
- PDCA 精力分配和周度复盘

待优化：
- AI 模型兼容性（当前绑定 Claude Code CLI）
- 图谱查询性能（grep 方案在节点增长后会遇到瓶颈）
- 多用户支持

### O（Output）— 认知代理进化

当前 `/brain` 是第一版——以用户的知识和思维方式对外输出，严格只读。

**实际使用案例**：用户粘贴一个朋友关于"AI 会不会取代人"的提问，`/brain` 会 grep 图谱找到用户在认知科学、AI 产品设计、教育等领域的 userInsight 和 ThinkingDeposit，然后以用户的认知风格和知识储备组织回答——不是通用的 AI 回复，而是"你本人会怎么说"。用户复制粘贴发给朋友即可。

进化方向：
- **多场景人格/语气适配**：工作、朋友、社交媒体等不同语境
- **长文生成**：从 Deposit 中编译完整文章，而非只做对话式输出
- **多格式**：文章、PPT 大纲、视频脚本、教学材料
- **发布管道**：直接连接发布平台（公众号、小红书等）
- **协作输出**：多人的认知图谱交叉，产出协作内容

## 需要协作推进的方向

### 近期（建立基础）

1. **AI 模型兼容**：当前绑定 Claude Code CLI，需要抽象出模型接口层，支持其他模型（GPT、Gemini 等）。第一步：梳理当前 Skill 中对 Claude 特有能力的依赖点，评估哪些是通用的、哪些需要适配层
2. **Skill 系统演进**：当前 Skill 是 Markdown 文件被 AI 读取执行（这是有意为之的去工程化设计），需要评估哪些场景确实需要代码化的 IO 层，哪些保持自然语言即可。第一步：列出当前所有 Skill 中调用 Bash/Python 的场景，分析哪些是"必须代码化的 IO"
3. **多用户架构**：当前单用户单文件夹，多用户需要考虑图谱隔离和共享。第一步：设计一个两用户的最小原型，验证文件夹隔离 + 共享图谱节点的可行性

### 中期（双向延伸）

4. **信号捕手产品化**（I 端）：信息源接入（RSS、社交媒体、Newsletter）→ 离心分层 → 预期构建 → 个性化呈现。第一步：选一个信息源（如 RSS）做端到端原型，验证"离心分层 → 按用户状态呈现"的完整链路
5. **输出能力进化**（O 端）：从对话式 brain 到结构化内容生成，场景化语气，发布管道。第一步：用现有 /brain + Deposit 内容尝试生成一篇完整文章，找到当前能力的瓶颈

### 远期（生态）

6. **图谱互联**：不同用户的图谱之间能否发现连接？（类似学术引用网络）
7. **垂直场景拓展**：教育（小学生学习路径）、健康（Coach 知识库）、项目管理等
