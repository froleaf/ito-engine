## 角色

知识考古学家。将用户过去积累的思考内容系统地导入知识图谱，还原认知发展的脉络。

## 使用场景

- 冷启动后，灌入过去数年的笔记和思考
- 日常使用中，偶然找到旧笔记想补录进系统
- 导入已发布的文章、公众号内容等历史产出

## 输入方式

支持三种方式，可混合使用：

1. **文件批量**：用户将文件放入 `inbox/bootstrap/`，Agent 逐篇处理
2. **对话直接输入**：用户在对话框中粘贴或口述历史内容
3. **链接**：用户提供 URL（如公众号文章链接），通过 WebFetch 读取

## 处理逻辑

### 核心原则

- **按用户给定的顺序处理**——用户会按思考的先后顺序输入（从早到晚）
- **图谱正常生长**——执行完整的 knowledge-extract + knowledge-process 流程
- **输出种子正常检测**——逐步识别和累积
- **不写 session_memory**——历史内容不是当下的交互
- **不触发 plan 更新**——灌入阶段不进入 PDCA 监测回路

### 对每份内容执行：

1. **理解内容**：阅读全文，理解核心主题和用户的思考脉络

2. **识别内容性质**：

   | 性质 | 判断依据 | 处理差异 |
   |------|----------|----------|
   | 学习笔记 | 引用外部来源为主 | 正常提取知识，创建 Material |
   | 思考记录 | 用户原创分析为主 | 重点提取 userInsight，创建 ThinkingDeposit |
   | 已发布产出 | 用户明确说"这是我发表的" | 创建 Deposit 实体标记 `outputStatus: "published"` |
   | 对话记录 | 多轮对话格式 | 提取用户发言中的知识点和见解 |

3. **知识提取**：执行 `/knowledge-extract` 的逻辑
   - 领域归属判断
   - 知识点提取与术语翻译
   - 与图谱匹配（create / deepen）
   - 跨域关联检测

4. **图谱更新**：执行 `/knowledge-process` 的逻辑，但有以下调整：
   - **正常执行**：图谱扩展、个人元数据标注、歧义检测、CrossDomainLink 创建、沉淀产出（Deposit）创建
   - **跳过**：session_memory 写入、plan 触发

5. **沉淀产出处理**（核心环节）：
   - 当内容构成**结构化的知识整理** → 创建 `KnowledgeDeposit`
   - 当内容构成**原创思考/模型** → 创建 `ThinkingDeposit`
   - 当内容是**已发布的文章** → 创建 Deposit 并标记 `outputStatus: "published"`
   - **查重合并**：先 grep 图谱中同领域/同主题的已有 Deposit，如果是对同一主题的延续或深化，合并到已有实体而非新建
   - 内容文件存入 `output/deposits/`，图谱实体中记录 `contentPath`
   - 已发布内容同时记录 `publishedUrl`（外部链接），但仍保留本地副本

6. **写入文件格式**：每个 deposit 文件必须包含 AI 导读 + 原文：

```markdown
# {产出标题}

> **类型**：知识沉淀 / 思考沉淀
> **状态**：沉淀 / 已发布（{渠道}）
> **协作**：与 {AI名称} 共同产出（如适用，否则省略此行）
> **涉及领域**：{领域列表}
> **关联节点**：{关键知识节点}
> **更新时间**：{最近更新日期}

## 导读

{AI 生成的内容摘要，约 200-500 字，
帮助用户快速回忆核心要点、论证结构和独特之处}

---

## 原文

{用户的完整原始内容，保留原始表达}
```

**查阅规则**：
- **Agent 自用**：只 Read 导读部分（到 `---` 为止），不读原文
- **用户问"这份讲了什么"**：基于导读回答
- **用户说"把原文给我"**：用 Bash 提取 `## 原文` 之后的内容直接打印，Agent 不读取
- **用户说"帮我分析原文"**：Agent 读取全文并消耗 token 分析

5. **归档**：将处理完的文件从 `inbox/bootstrap/` 移到 `inbox/processed/`

### 输出潜力标记

对尚未成型但有输出潜力的想法，不创建 Deposit，而是在相关图谱节点上标记 `outputPotential: "high"`。Agent 可通过 grep `outputPotential` 动态汇总候选列表，用户想写东西时直接查询即可。

### 批量处理时的节奏

**逐篇处理，禁止批量并行**：无论有多少篇内容，必须一篇一篇顺序执行完整的 knowledge-extract → knowledge-process 流程。不得为了效率将多篇内容并行处理或跳过 knowledge-process 的标准模板。批量并行会导致实体字段遗漏（如用 `domains` 代替 `aboutNodes`/`aboutDomains`，造成图谱节点无边连接）。

- 每处理 3-5 篇，给用户一个简要进度反馈
- 如果发现新领域候选，暂停询问用户是否注册
- 如果发现重要的跨域关联或输出种子，即时告知
- 所有内容处理完后，执行 `python3 scripts/build_visualization.py` 同步可视化
- 生成一份**灌入摘要**存入 `inbox/conversations/`

### 灌入摘要格式

```markdown
# 历史内容灌入摘要

- 日期：{YYYY-MM-DD HH:MM}
- 处理数量：{N} 篇
- 涉及领域：{领域列表}

## 图谱变化
- 新增知识节点：{N} 个
- 深化已有节点：{N} 个
- 新增跨域关联：{N} 条

## 输出种子
- 新识别种子：{列表}
- 已发布产出：{列表}

## 关键发现
{值得注意的跨域关联、用户理论等}
```

## 与 /cold-start 的配合

冷启动认识对话完成后，Agent 应提示：
- "骨架已搭好。如果你有历史笔记想灌入，可以用 `/bootstrap`。灌入完成后再用 `/plan` 生成首次计划。"

## 与日常使用的配合

系统正式启用后，用户随时可以 `/bootstrap` 补录旧内容：
- 图谱和输出种子正常更新
- 不影响当前 session_memory 和 plan 的统计
